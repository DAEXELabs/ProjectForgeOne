# Forge Journal

The Forge Journal is the observation system for Forge One Infrastructure.

Its purpose is to capture real business friction before solutions are designed. The journal helps separate interesting ideas from repeated, measurable problems that deserve product or automation investment.

## Purpose

The Forge Journal exists to:

- Record operational friction as it happens
- Capture evidence before memory fades
- Identify repeated problems across people, teams, customers, and workflows
- Prevent premature product design
- Create a clear path from observation to validated Forge Service

## How Observations Become Forge Services

Forge Services should not begin as assumptions. They should begin as observed friction.

The path is:

1. Observation: Something inefficient, risky, repetitive, slow, or unclear happens.
2. Journal Entry: The event is documented using the journal template.
3. Evidence: Similar events are counted and compared over time.
4. Validation: A pattern becomes a validated business problem.
5. Opportunity Register: The problem is added or updated in the register.
6. Prototype: A small service or automation is considered only after the problem is clear.
7. Implementation: A service becomes part of Forge One Infrastructure only after it proves value.

## How to Write Entries

Use [Template.md](./Template.md) for each entry.

Create entries inside the folder for the current year:

```text
docs/journal/2026/
```

Recommended file name format:

```text
YYYY-MM-DD-short-description.md
```

Write entries in plain language. The goal is to describe what happened, how it is handled today, what friction exists, and what business impact might exist.

Do not over-design the solution inside the journal entry. The journal captures reality first.

## Weekly Review Process

Complete one weekly review at the end of each week.

During review:

- Read all new journal entries from the week
- Group similar observations
- Update evidence counts in the Opportunity Register
- Mark problems as validated only when a pattern is visible
- Update the Scoreboard
- Decide the next action for each meaningful opportunity

Weekly reviews should answer:

- What friction repeated this week?
- What cost time, money, quality, leadership attention, or customer trust?
- What is still just an isolated observation?
- What is ready for deeper validation?
- What should not be pursued yet?

## Status Definitions

- Observed: Captured once or only lightly evidenced
- Validated: Repeated or supported by measurable evidence
- Prototype: Approved for a small test solution
- Implemented: Solved by an internal automation or Forge Service
